DevLogger centralises error reporting so no team reinvents it. Here’s exactly how a failure flows through the system — and where it quietly saves your developers real time.

The journey

What happens on one click

Five stages, most of which your team never has to build or maintain.

Any internal module

Something fails

An API call times out, a save returns 500, a page can’t load. The module catches it and calls DevLoggerModal.showError() with a friendly message and the raw context.

The user

Sees one clear modal

A calm dialog appears: a plain-language message, “Mengerti” to dismiss, and “Laporkan ke IT” to report. No stack traces shouted at the user — those hide under “Detail teknis”.

The widget

Posts the full context

On Laporkan ke IT, the widget sends everything — URL, method, status, stack, request body, user — to POST /api/v1/errors/report, authenticated with the user’s LMS token.

DevLogger server

Saves & emails automatically

The server persists a row to MSSQL logging_mike and sends a full-context alert email to the IT recipients via Nodemailer — both in a single request.

IT / developers

Get everything to reproduce it

The team receives a reference number and an email with the exact endpoint, status, stack, and payload — no back-and-forth screenshots. The user sees a “terkirim” confirmation.

Why it helps

The developer time it quietly buys back

😖 Without DevLogger

  • Every module rolls its own error dialog and logging.
  • Bug reports arrive as “it’s broken” with a blurry screenshot.
  • Devs guess the endpoint, status, and payload.
  • Errors live in scattered logs — or nowhere at all.
  • No consistent UX; users don’t know how to report.

✅ With DevLogger

  • One shared widget — nothing to rebuild per app.
  • Reports carry the full stack, URL, status & payload.
  • Every error is saved to logging_mike and searchable.
  • IT is emailed instantly with a reference number.
  • Identical, friendly UX in every module.

Under the hood

What the server already handles for you

Auth & rate limiting

LMS-token auth on every route plus a stricter 10 req / 15 min limit on the report and email endpoints.

MSSQL persistence

Structured rows in logging_mike — queryable by status, date, and free-text search.

Alert emails

Nodemailer sends a full-context HTML alert to the IT list — with graceful fallback when SMTP is down.
